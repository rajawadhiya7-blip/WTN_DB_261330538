Normalization Mini Project 



Question 1 - Normalize into 1st Normal Form (1NF)

Original Table:
Member_Id | First_Name | Last_Name | Hobbies
101       | Jayson     | Mark      | Cricket, Swimming, Football
102       | Ram        | Ganesh    | Swimming, Running, Music
103       | Raj        | Kishore   | Dancing, Singing, Running

Problem:
The Hobbies column has multiple values in a single cell separated
by commas. This violates 1NF because 1NF requires all values to
be atomic (one value per cell).

Solution - split each hobby into a separate row:

Member_Id | First_Name | Last_Name | Hobbies
101       | Jayson     | Mark      | Cricket
101       | Jayson     | Mark      | Swimming
101       | Jayson     | Mark      | Football
102       | Ram        | Ganesh    | Swimming
102       | Ram        | Ganesh    | Running
102       | Ram        | Ganesh    | Music
103       | Raj        | Kishore   | Dancing
103       | Raj        | Kishore   | Singing
103       | Raj        | Kishore   | Running

Now each cell has only one value so the table is in 1NF.


Question 2 - Normalize into 2nd Normal Form (2NF)

Original Table:
Empno | Training   | Training_Date | Dept
101   | Oracle SQL | 12-Aug-2015   | TT
101   | Java       | 21-Aug-2015   | BU
102   | Oracle SQL | 18-Sep-2014   | TT

Problem:
The composite primary key is (Empno, Training).
- Training_Date depends on both Empno and Training (full dependency) - ok
- But Dept depends only on Training, not on Empno.
  Oracle SQL is always TT and Java is always BU regardless of employee.
This is a partial dependency which violates 2NF.

Solution - split into two tables:

Table 1: Employee_Training
Empno | Training   | Training_Date
101   | Oracle SQL | 12-Aug-2015
101   | Java       | 21-Aug-2015
102   | Oracle SQL | 18-Sep-2014
Primary Key: (Empno, Training)

Table 2: Training_Department
Training   | Dept
Oracle SQL | TT
Java       | BU
Primary Key: Training

Now every non-key attribute depends on the full primary key so
the tables are in 2NF.


Question 3 - Normalize into 3rd Normal Form (3NF)

Original Table:
Member_Id | First_Name | Last_Name | Sports   | Fees
101       | Rajesh     | Chand     | Cricket  | 100
102       | Jayesh     | Raj       | Hockey   | 80
103       | Mark       | Dorson    | Foodball | 90

Problem:
Primary key is Member_Id.
- First_Name, Last_Name, Sports depend directly on Member_Id - ok
- But Fees depends on Sports, not on Member_Id.
  Cricket always costs 100, Hockey always costs 80.
So the dependency chain is: Member_Id -> Sports -> Fees
This is a transitive dependency which violates 3NF.

Solution - split into two tables:

Table 1: Member_Sports
Member_Id | First_Name | Last_Name | Sports
101       | Rajesh     | Chand     | Cricket
102       | Jayesh     | Raj       | Hockey
103       | Mark       | Dorson    | Foodball
Primary Key: Member_Id
Foreign Key: Sports

Table 2: Sports_Fees
Sports   | Fees
Cricket  | 100
Hockey   | 80
Foodball | 90
Primary Key: Sports

Now there are no transitive dependencies so the tables are in 3NF.
